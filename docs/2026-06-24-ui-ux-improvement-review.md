## 1. 확인 기준

로컬 기준 브랜치는 `chore/design-review`이며, 로컬 HEAD는 `dd231ba7e118841081e8d9d78c49c9c94a06a667`입니다. 이 커밋은 원격 `develop` HEAD와 같습니다. 원격 기본 브랜치인 `main` HEAD는 `436ab3f9868323f70c1b718bde6a7170efec80be`로 확인되어, 저장소 기본 HEAD와 현재 로컬 검토 브랜치가 다릅니다.

GitHub 저장소 URL은 HTTP 200으로 접근 가능했습니다. 배포 페이지 HTML도 HTTP 200으로 응답했고, 현재 로컬 `index.html`과 같은 핵심 랜딩 구조를 렌더링하고 있었습니다.

브라우저 확인은 다음 조건에서 수행했습니다.

- 데스크톱: 1280 x 720
- 모바일: 390 x 844
- 확인 항목: 루트 진입, `#how-it-works` 해시 진입, 모바일 메뉴, 주요 섹션 위치, 콘솔 오류
- 결과: 확인 중 콘솔 error/warn은 보이지 않았습니다. 외부 제출이나 실제 개인정보 입력은 수행하지 않았습니다.

## 2. 로컬 구조 요약

현재 프로젝트는 작은 정적 랜딩 페이지와 Cloudflare Pages Functions 기반 AI 인터뷰 API가 함께 있는 구조입니다.

- `index.html`: 랜딩 전체 구조, 네비게이션, 히어로, Before/After 예시, 진행 방식, AI 인터뷰 섹션, FAQ, Turnstile 설정
- `styles.css`: 다크 터미널 톤의 반응형 UI, 카드, 타임라인, 인터뷰 UI 스타일
- `src/app.js`: 인터뷰 UI 초기화, 모바일 메뉴, 스크롤 페이드, 파티클 배경
- `src/interview-ui.js`: 온페이지 AI 인터뷰 상태 기계, 이메일 게이트, A/B 답변 비교, 평점 저장
- `src/interview-client.js`: `/api/interview/turn`, `/api/interview/compare` 호출 및 SSE 파싱
- `src/form-utils.js`: 이메일/동의 검증, 민감정보 감지, 인터뷰 payload 생성
- `functions/api/interview/*.js`: Cloudflare Pages Functions 인터뷰 API
- `apps-script/Code.gs`: Google Sheets 저장 파이프라인
- `tests/`: 유틸, 클라이언트, Functions 단위 테스트
- `docs/superpowers/`: AI QA 인터뷰 설계 및 빌드 계획 문서

## 3. 현재 UX 진단

현재 페이지의 가장 큰 장점은 "설명형 랜딩"에서 끝나지 않고 실제 AI 인터뷰를 바로 체험하게 만든다는 점입니다. 개발자 대상 서비스에 맞춘 터미널 콘셉트, Java/Spring 맥락, 일반 답변과 맥락 반영 답변의 비교 예시는 제품 가설을 빠르게 전달합니다.

다만 전환 UX 관점에서는 행동 지점이 늦습니다. 데스크톱 루트 진입 기준 인터뷰 섹션은 첫 화면 아래 약 1,250px 지점에 있고, 모바일 루트 진입 기준 인터뷰 섹션은 약 1,800px 아래에 있습니다. 히어로에는 "5분만 들려주세요"라는 명확한 요청이 있지만 즉시 인터뷰로 이동하는 주요 CTA가 없습니다.

또 하나의 중요한 리스크는 첫 입력 전에 AI 처리와 외부 전송에 대한 충분한 동의 맥락이 보이지 않는다는 점입니다. 현재 이메일 게이트에서 수집/AI 처리 동의 문구가 나오지만, 사용자의 첫 답변은 그 전에 `/api/interview/turn`으로 전송될 수 있습니다. 플레이스홀더로 "코드/로그/URL 없이 상황만 적어주세요"가 있기는 하지만, 개인정보와 민감정보를 다루는 신뢰 UX로는 약합니다.

`#how-it-works` 해시 진입은 동작하지만, 고정 네비게이션 높이 61px에 비해 타깃 섹션의 상단이 y=44 부근에 위치해 섹션 상단 일부가 네비게이션 뒤에 걸칠 수 있습니다. `scroll-padding-top` 또는 섹션별 `scroll-margin-top`을 조정해야 합니다.

모바일 메뉴는 햄버거 버튼으로 정상 열리고 닫히는 구조입니다. 모바일 레이아웃 자체도 큰 깨짐은 없지만, 히어로 터미널과 Before/After 예시가 길어 실제 인터뷰 시작까지의 거리가 길어집니다.

## 4. 우선순위별 개선 방향

### P0. 첫 입력 전 신뢰와 동의 UX 보강

첫 인터뷰 입력 전에 최소한의 안내와 동의 상태를 명확히 해야 합니다.

권장 변경:

- 첫 질문 아래에 짧은 안내를 추가합니다: "코드, 로그, URL, 비밀값은 입력하지 마세요. 입력한 내용은 AI 질문 생성을 위해 처리됩니다."
- 첫 전송 버튼 위 또는 인터뷰 시작 카드에 필수 체크박스를 둡니다. 문구는 이메일 저장 동의와 분리해 "AI 인터뷰 처리를 위한 입력 전송 동의"로 좁힙니다.
- 민감정보 차단은 현재 유틸/서버에서 수행하되, 차단 시 메시지는 더 친절하게 바꿉니다. 예: "보안을 위해 URL/코드/로그는 받을 수 없어요. 상황만 한두 문장으로 다시 적어주세요."
- 이메일 게이트의 동의 문구는 "저장 및 베타 연락 동의"로 역할을 좁히고, 첫 입력 전 AI 처리 동의와 중복되지 않게 정리합니다.

기대 효과: 사용자가 안심하고 입력할 수 있고, 실제 AI 호출 전에 동의 UX가 맞춰집니다.

### P1. 히어로에서 인터뷰 시작까지의 거리 단축

현재 히어로는 메시지는 좋지만 행동 버튼이 없습니다. 랜딩의 목표가 "완료 리드 20명"이라면 첫 화면에서 인터뷰 시작으로 강하게 안내해야 합니다.

권장 변경:

- 히어로 하단에 주요 CTA를 추가합니다: "AI 진단 시작하기"
- 보조 CTA는 "진행 방식 보기" 정도로 제한합니다.
- CTA 클릭 시 `#apply`로 스크롤하되, 고정 네비게이션에 가리지 않도록 `scroll-margin-top`을 적용합니다.
- 모바일에서는 히어로 바로 아래에 압축형 인터뷰 시작 프롬프트를 배치하거나, `#apply` 섹션을 Before/After와 진행 방식보다 앞으로 이동하는 A/B 테스트를 고려합니다.
- 인터뷰 섹션 헤드라인을 "AI 학습 진단 인터뷰"보다 행동 중심으로 바꿉니다. 예: "막혔던 순간을 한 문장으로 시작하세요"

기대 효과: 방문자가 "읽는 페이지"에서 "바로 해보는 페이지"로 더 빨리 전환됩니다.

### P1. 해시 이동과 고정 네비게이션 정렬 수정

`#how-it-works` 진입 시 섹션 상단이 네비게이션에 살짝 덮일 수 있습니다.

권장 변경:

- `html { scroll-padding-top: 84px; }` 정도로 상향 조정합니다.
- 주요 앵커 섹션에 `scroll-margin-top: 84px;`를 명시합니다.
- 모바일/데스크톱에서 `#hero`, `#how-it-works`, `#apply`, `#faq` 링크 클릭 후 제목이 네비게이션 아래에 온전히 보이는지 확인합니다.

기대 효과: 공유 링크와 네비게이션 클릭의 첫 인상이 더 안정적입니다.

### P1. 모바일 첫 화면 압축

모바일에서는 히어로 터미널이 662px 높이로, 그 뒤 Before/After와 진행 방식까지 지나야 인터뷰가 보입니다. 첫 화면의 감성은 좋지만 전환에는 부담입니다.

권장 변경:

- 모바일에서 터미널 바, 프롬프트 라인, 칩 영역의 수직 간격을 줄입니다.
- Before/After 예시는 모바일에서 접힘형 또는 1개 강조 카드로 축약합니다.
- 진행 방식은 3단계 전체 설명보다 아이콘/짧은 문장 중심으로 줄입니다.
- `AI 진단 시작하기` 버튼을 모바일 하단 sticky CTA로 제공하되, 인터뷰 섹션에 도착하면 숨깁니다.

기대 효과: 모바일 사용자가 "읽다가 이탈"하기 전에 입력 행동에 도달합니다.

### P2. 인터뷰 진행감과 오류 회복 강화

현재 인터뷰는 간단하고 빠르지만, 사용자가 어느 단계에 있는지와 실패 시 무엇을 하면 되는지가 더 드러나야 합니다.

권장 변경:

- 인터뷰 로그 상단에 `질문 1/3`, `답변 비교 생성 중`, `결과 평가` 같은 단계 표시를 추가합니다.
- "생각 중..." 상태에는 예상 대기 문구를 붙입니다. 예: "보통 5~10초 걸립니다."
- `/turn` 실패와 `/compare` 실패를 구분해 재시도 버튼을 제공합니다.
- A/B 답변 카드의 동일한 버튼 문구에는 접근성 라벨을 붙입니다. 예: `aria-label="답변 1이 더 유용함 선택"`.
- 답변을 선택하지 않으면 제출 버튼을 비활성화하고, 선택 후에는 카드 선택 상태와 버튼 텍스트를 명확히 바꿉니다.

기대 효과: AI 응답 지연과 오류가 전환 실패로 이어질 확률을 줄입니다.

### P2. 카피와 정보 구조 정리

현재 카피는 개발자 콘셉트가 강하고 일관적입니다. 다만 주니어/취준 사용자가 얻는 결과가 첫 화면에서 조금 더 직접적으로 보여야 합니다.

권장 변경:

- 히어로 보조문구에 산출물을 명시합니다. 예: "마지막에 내 상황 기준 답변과 일반 답변을 비교하고, 다음 2주 학습 방향을 확인합니다."
- "100인 베타 우선 초대"는 보상으로 좋지만, 사용자가 즉시 받는 가치인 "맥락 답변 비교"를 더 앞에 둡니다.
- Evidence 카드의 `#` 장식은 개발자 감성에는 맞지만 보조 정보처럼 보일 수 있으므로, 결과 중심 제목으로 정리합니다.
- FAQ에 "입력하면 안 되는 정보"를 별도 항목으로 추가합니다.

기대 효과: 기술 콘셉트는 유지하면서 초보자도 망설임 없이 참여할 수 있습니다.

### P2. 시각 디자인과 브랜드 확장성

터미널 기반 다크 UI는 타깃 개발자에게 빠르게 맥락을 전달합니다. 하지만 장기적으로는 "개발자 도구" 느낌이 강해 학습 코치/진단 서비스의 신뢰감과 사람 냄새가 약해질 수 있습니다.

권장 변경:

- 현재 팔레트는 유지하되, 핵심 행동 영역에는 더 높은 대비와 따뜻한 보조 색을 제한적으로 추가합니다.
- 파티클 배경은 reduced motion을 존중하고 있어 좋습니다. 다만 모바일에서는 제거하거나 opacity를 낮춰 성능과 집중도를 확보합니다.
- 카드 반경, 입력 높이, 섹션 간격, 버튼 상태를 디자인 토큰화해 이후 랜딩 확장 시 일관성을 확보합니다.
- OG 이미지 또는 공유 썸네일을 추가해 GitHub, Slack, 카카오톡 공유 시 첫인상을 개선합니다.

기대 효과: 실험용 페이지에서 제품형 랜딩으로 확장하기 쉬워집니다.

### P2. 측정 이벤트 설계

README의 Round 1 기준은 명확하지만, 현재 UI 개선 판단에 필요한 중간 퍼널 이벤트가 더 필요합니다.

권장 이벤트:

- `landing_view`: variant, utm, referrer, viewport
- `hero_cta_click`
- `interview_first_input_focus`
- `interview_first_turn_submit`
- `turn_error`
- `email_gate_view`
- `email_gate_submit`
- `compare_start`
- `compare_done`
- `answer_selected`
- `rating_submit`
- `save_error`

기대 효과: 단순 완료 수뿐 아니라 어디서 막히는지 확인해 다음 실험을 더 빠르게 설계할 수 있습니다.

## 5. 추천 실행 순서

1일 내 적용:

- 히어로 CTA 추가 및 `#apply` 이동
- `scroll-margin-top`/`scroll-padding-top` 보정
- 첫 입력 전 민감정보/AI 처리 안내 추가
- FAQ에 "입력하면 안 되는 정보" 추가

2~3일 내 적용:

- 모바일 히어로 압축 및 sticky CTA 실험
- 인터뷰 단계 표시, 대기 상태, 재시도 UX 개선
- A/B 카드 선택 접근성 라벨과 선택 상태 개선
- 퍼널 이벤트 로깅 설계 및 최소 구현

다음 라운드:

- 모바일 우선 랜딩 순서 A/B 테스트: `Hero -> Interview -> Before/After -> How it works`
- 카피 A/B 테스트: "5분만 들려주세요" vs "막힌 순간을 입력하면 맞춤 답변을 비교해드립니다"
- 공유용 OG 이미지와 신뢰 요소 추가

## 6. 결론

현재 배포 페이지는 제품 가설을 실제 체험으로 연결한다는 점에서 좋은 출발점입니다. 가장 먼저 고칠 부분은 더 화려한 UI가 아니라 전환과 신뢰입니다. 첫 화면에서 바로 시작할 수 있게 만들고, 첫 입력 전에 AI 처리/민감정보 안내를 명확히 하면 Round 1 검증의 완료율과 데이터 품질이 함께 좋아질 가능성이 큽니다.
